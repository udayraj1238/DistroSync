System design

Three roles. One protocol.

Every component communicates over TCP using a 4-byte length-prefixed JSON protocol. Producers submit tasks, the broker queues and manages them, workers execute and acknowledge. Zero external dependencies.

What's under the hood

Six concepts. Built, not borrowed.

Every mechanism was written from scratch to understand exactly how production queuing infrastructure works at a systems level.

⚡
asyncio event loop
A single thread handles thousands of concurrent TCP connections without blocking. The broker processes messages using Python's cooperative multitasking — no thread overhead, no context-switching cost.
broker/server.py
⚙️
ProcessPoolExecutor
Python's GIL prevents true CPU parallelism with threads. Workers bypass this by running tasks in separate OS processes. Real parallelism, not concurrent waiting — each worker uses a full CPU core.
worker/task_executor.py
🪣
Adaptive token bucket
The fill rate dynamically adjusts based on live queue depth and worker latency. Under 5× overload the bucket slows to 5% of normal — the system throttles itself rather than crashing.
broker/load_shedder.py
💾
SQLite WAL mode
Write-Ahead Logging appends all task state changes to a log before the database. On crash restart the WAL replays automatically — zero tasks lost across 10 simulated crashes in testing.
persistence/wal_store.py
💓
Heartbeat eviction
Workers send a heartbeat every 2 seconds. After 3 missed heartbeats the broker evicts the worker and re-queues all in-flight tasks. The same mechanism Kubernetes and ZooKeeper use to detect node failure.
broker/worker_registry.py
📬
Dead letter queue + replay
Tasks that fail 3 consecutive times move to the DLQ instead of being silently dropped. An admin can inspect them, fix the root cause, and replay via the REST API — exactly how AWS SQS and RabbitMQ work.
broker/dead_letter.py
